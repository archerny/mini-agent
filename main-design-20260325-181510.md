## Positioning

> **mini-agent 是 AI Agent 网络的实时操作系统。**

一个自研的多 Agent 协调引擎（Go Runtime）+ 实时可视化作战大屏（React），
用于观测、理解和管理动态 Agent 协作网络。终极愿景：一人超级公司——
给一个高层目标，系统自动分解、分配、执行、交付、复盘。

## Problem Statement

现有的 AI 工作流平台（n8n / Dify / Coze）本质上是静态 DAG 流水线——节点按预设路径执行，没有真正的多 Agent 动态协作。当你想构建一个"一人超级公司"——多个 AI Agent 扮演不同角色自主协作完成复杂任务——现有工具无法支撑。你需要：

1. **Agent Runtime**：一个真正的多 Agent 协调引擎，Agent 之间是动态的、有状态的、自主的交互
2. **可视化管理平台**：一个实时作战指挥大屏，看到每个 Agent 的状态、消息流、依赖关系

### 具体痛点

- 看不到 Agent 在做什么——多个 Agent 并行运行时，无法直观了解每个 Agent 的当前状态
- 消息流不可追踪——Agent 之间传了什么消息、顺序是什么、有没有丢失，完全是黑盒
- 无法定位瓶颈——任务卡在哪个 Agent 上？谁在等谁？没有全局视图
- 无法动态调整——现有流水线是写死的 DAG，运行时无法让 Agent 自主决定协作方式

## What Makes This Cool

**核心差异化：从"流水线"到"协作网络"**

- 现有工具：节点 A → 节点 B → 节点 C（预设路径，静态执行）
- Mini-Agent：Agent A ↔ Agent B ↔ Agent C（动态交互，运行时决策）

可视化大屏不是看一个线性进度条，而是看一个**活的网络**——消息在节点间实时流动，能看到谁在等谁、谁卡住了、谁完成了、谁在跟谁对话。像一个真正的公司作战室。

**差异化 Demo**：运行时一个 Agent 主动拉另一个 Agent 加入协作——这在 n8n/Dify 里不可能做到。

终极愿景：给一个高层目标，系统自动分解、分配、执行、交付、复盘。

## Constraints

- 自用优先，不追求通用性
- 第一步是"看得见"（实时监控大屏），再做"管得了"（管理控制台）
- 技术选型：Go Runtime + React 前端（Protocol 优先方法论）
- 单人维护，架构要简洁可控

## Anti-Scope (MVP 明确不做)

- ❌ Agent 管理/创建 UI（MVP 只读，不提供 CRUD 操作界面）
- ❌ 认证/权限系统（本地自用，不需要多用户）
- ❌ 持久化存储（MVP 纯内存，重启清零）
- ❌ 真实 LLM 调用（MVP 用 mock/模拟 Agent 验证架构）
- ❌ Agent 配置文件格式设计（MVP 用硬编码 demo 场景）
- ❌ 多 Runtime 实例 / 分布式部署
- ❌ 国际化 / 多语言

## Premises

1. ✅ 核心差异化是「真正的多 Agent 动态协作」而非「静态 DAG 流水线」
2. ✅ 自研 Agent Runtime，不套壳现有框架，对协调机制有完全控制权
   — 但后续可以引入已有框架作为 Agent 的 LLM 调用层（如调用 OpenAI SDK）
3. ✅ 第一步是「看得见」（实时监控大屏），而不是「管得了」
   — Demo 场景要足够真实，不是互相说 hello，而是完成一个实际小任务
4. ✅ Web 页面和 Runtime 是两个独立但耦合的系统，通过协议通信
5. ✅ 自己是第一个用户，先解决自己的真实痛点
6. ✅ **Protocol 优先**——先设计 Agent 通信协议，再构建 Runtime 和前端
   — 协议是核心资产，Runtime 和前端都可以独立替换
   — 任何语言（Python/JS/Go）的 Agent 都能通过协议加入网络

## Agent Communication Protocol (核心协议设计)

### 设计原则

- **Protocol 是系统的宪法**——所有组件围绕协议构建
- **JSON-based**——人可读，易调试，生态友好
- **事件驱动**——所有状态变更通过事件传播
- **可扩展**——预留 metadata/extensions 字段
- **MessageBus 是唯一事件源 (Single Source of Events)**——Agent 只管发消息，MessageBus 拦截后自动产生 sent/received/error 事件。Agent 代码不感知 EventStream，完全解耦。

### Agent 生命周期状态

```
spawning → ready → idle ←→ busy → completed → idle
  ↓                 ↓        ↓
 error            error    error
                    ↓        ↓
                 shutdown  (recoverable → idle)
                           (non-recoverable → shutdown)
```

完整状态转换表：

| 当前状态 | 可转换到 | 触发条件 |
|---------|---------|---------|
| `spawning` | `ready` | 初始化成功 |
| `spawning` | `error` | 初始化失败（超时/配置错误/panic） |
| `ready` | `idle` | 自动转换（ready 是瞬态，表示"初始化完成，首次进入空闲"） |
| `idle` | `busy` | 收到消息/任务 |
| `idle` | `error` | 内部错误 |
| `idle` | `shutdown` | 收到关闭指令 |
| `busy` | `completed` | 当前任务完成 |
| `busy` | `error` | 任务执行出错 |
| `completed` | `idle` | 自动转换（completed 是瞬态，供事件系统捕获） |
| `error` | `idle` | 可恢复错误，Agent 自愈或被重置 |
| `error` | `shutdown` | 不可恢复错误 |
| `shutdown` | (终态) | Agent 已关闭，不可再转换 |

### 消息格式

```json
{
  "id": "msg_uuid_v7",
  "type": "agent.message | agent.request | agent.response | agent.broadcast",
  "from": "agent_id",
  "to": "agent_id | *",
  "correlation_id": "original_msg_id (用于 request/response 配对)",
  "timestamp": "2026-03-25T18:15:10.000Z",
  "payload": {
    "content_type": "text | json | tool_call | tool_result",
    "content": "..."
  },
  "metadata": {}
}
```

### 消息类型

| 类型 | 用途 |
|------|------|
| `agent.message` | Agent 间的普通消息 |
| `agent.request` | Agent 向另一个 Agent 发起请求，期望回复 |
| `agent.response` | 对 request 的回复，通过 correlation_id 关联 |
| `agent.broadcast` | Agent 向所有 Agent 广播消息 |

### Agent 事件（系统级）

事件由 **MessageBus** 和 **Agent Manager** 自动产生，Agent 代码不直接 emit 事件。

```json
{
  "id": "evt_uuid_v7",
  "type": "event_type",
  "sequence": 12345,
  "agent_id": "agent_id",
  "timestamp": "2026-03-25T18:15:10.000Z",
  "data": { "...事件具体数据..." },
  "metadata": {}
}
```

**新增 `sequence` 字段**：全局递增序列号，用于前端重连时检测 gap 并补偿。

| 事件类型 | 产生者 | 触发时机 | data 内容 |
|---------|--------|---------|----------|
| `agent.spawned` | Agent Manager | Agent 创建并初始化完成 | `{name, role, capabilities}` |
| `agent.state_changed` | Agent Manager | Agent 状态变更 | `{prev_state, new_state, reason}` |
| `agent.message_sent` | MessageBus | Agent 发送消息（MessageBus 拦截后产生） | `{message}` |
| `agent.message_received` | MessageBus | Agent 接收消息（MessageBus 投递后产生） | `{message}` |
| `agent.error` | MessageBus / Agent Manager | Agent 发生错误 | `{error, stack, recoverable}` |
| `agent.shutdown` | Agent Manager | Agent 关闭 | `{reason, exit_code}` |
| `network.topology_changed` | MessageBus | 网络拓扑变更 | `{change_type: "agent_joined|agent_left|link_created|link_removed", details}` |

**注意**：`agent.message_sent` 和 `agent.message_received` 的 data 字段包含完整的 Message 对象。MVP 接受数据冗余，保持简单。后续可改为只包含 `message_id` 引用。

### Agent 能力声明 (Agent Card)

```json
{
  "id": "agent_id",
  "name": "researcher",
  "role": "Research Agent — 负责信息搜集和分析",
  "capabilities": ["web_search", "summarize", "analyze"],
  "accepts": ["agent.request", "agent.message"],
  "status": "idle",
  "metadata": {
    "model": "gpt-4",
    "max_concurrent_tasks": 3
  }
}
```

### 消息处理规则 (MessageBus)

| 场景 | 处理方式 |
|------|---------|
| 正常点对点消息 | 路由到目标 Agent 消息队列，emit `message_sent` + `message_received` |
| 广播消息 (`to: "*"`) | fan-out：遍历所有 Agent 逐一投递，每个都 emit `message_received` |
| 目标 Agent 不存在 | 返回 error，emit `agent.error`（type=undeliverable） |
| 目标 Agent 已 shutdown | 返回 error，emit `agent.error`（type=undeliverable） |
| 目标 Agent 消息队列满 | 丢弃消息，emit `agent.error`（type=backpressure） |
| 消息格式不合法 | 拒绝，emit `agent.error`（type=invalid_message） |
| 消息 payload 超大 | 拒绝（上限 1MB），emit `agent.error`（type=payload_too_large） |
| 广播风暴 | 限流：单 Agent 每秒最多 10 条广播 |
| `agent.response` 的 correlation_id 无效 | 视为普通消息处理（不阻断） |

## Approaches Considered

### Approach A: Go Runtime + React（原设计文档方案，未选）

直接开始做 Runtime 引擎，协议后面再定。

- Pros: 更快能跑起来
- Cons: 协议是事后补丁，容易导致 Runtime 和前端耦合
- Effort: L
- Completeness: 7/10

### Approach B: Go Runtime + 嵌入式前端（HTMX，未选）

用 Go embed + HTMX 替代 React，追求真正的单二进制。

- Pros: 极致简洁，零 Node.js 依赖
- Cons: HTMX 做不了复杂的 Agent 网络拓扑可视化
- Effort: M
- Completeness: 5/10

### Approach C: Go Runtime + React + Protocol 优先（✅ 选中）

先设计 Agent 通信协议，然后 Runtime 和前端都围绕协议构建。

- Pros: 协议是核心可复用资产；任何语言的 Agent 可接入；前端天然知道怎么渲染
- Cons: 前期多花时间设计协议（但成本很低）
- Effort: L
- Completeness: 9/10

## Recommended Approach

**Go Runtime + React 前端（Protocol 优先）**

### 架构概览

```
┌──────────────────────┴──────────────────────────────┐
│                   Go Backend                         │

│  │           Agent Runtime Engine                │   │
│  │  ┌─────────┐  ┌──────────┐  ┌────────────┐  │   │
│  │  │ Agent   │  │ Message  │  │ Event      │  │   │
│  │  │ Manager │  │ Bus      │  │ Stream     │  │   │
│  │  │         │  │ (event   │  │            │  │   │

│  │  ┌──────────────────────────────────────┐    │   │
│  │  │  Agent structs (state + goroutine)   │    │   │
│  │  │  Agent A    Agent B    Agent C       │    │   │
│  │  └──────────────────────────────────────┘    │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**关键设计决策**：
1. **Agent = struct + goroutine**（而非 Agent = goroutine）——Agent 可以休眠/恢复/迁移
2. **MessageBus = Single Source of Events**——Agent 不感知事件系统，完全解耦
3. **WebSocket 单向推送**（MVP）——预留双向 command channel 能力

### 核心模块

#### 1. Agent Communication Protocol（核心协议层）

- **消息格式**: 统一的 JSON 消息协议（见上方协议设计）
- **事件系统**: MessageBus + Agent Manager 自动产生事件，Agent 不感知
- **Agent Card**: Agent 能力声明，供发现和路由

#### 2. Agent Runtime Engine（Go）

- **Agent struct**: 有状态的 Agent 实体，包含 ID、角色、状态机、消息队列
- **goroutine executor**: Agent 的执行引擎，负责处理消息队列中的消息。`recover()` 捕获 panic。
- **Message Bus**: 基于协议的消息路由，支持点对点、请求/响应、广播。**唯一的事件产生者**（消息相关事件）。
- **Event Stream**: 所有事件的有序流（带 sequence number），供 API 层推送给前端
- **Agent Manager**: 管理 Agent 生命周期。**唯一的生命周期事件产生者**。

#### 3. API Layer（Go）

- **Protocol WebSocket** (`/ws/events`): 将 Event Stream 实时推送给前端（单向）
- **REST API**: 获取当前网络快照、Agent 列表、消息历史
- **CORS + 开发代理**: 支持前后端独立开发

##### REST API 端点

| Method | Path | 用途 | 响应 |
|--------|------|------|------|
| GET | `/api/agents` | 获取所有 Agent 列表（含 Agent Card） | `AgentCard[]` |
| GET | `/api/agents/:id` | 获取单个 Agent 详情 | `AgentCard` |
| GET | `/api/messages` | 获取消息历史（支持 `?limit=N&before=timestamp`） | `Message[]` |
| GET | `/api/events` | 获取事件历史（支持 `?since_sequence=N&limit=N`） | `Event[]` |
| GET | `/api/topology` | 获取当前网络拓扑（邻接表） | `{nodes: AgentCard[], edges: {from, to, message_count}[]}` |
| GET | `/api/stats` | 获取全局统计 | `{agent_count, message_count, active_agents, error_count, uptime}` |
| WS | `/ws/events` | 实时事件推送 | Event stream (JSON per line) |

##### WebSocket 协议

- **连接**: `ws://localhost:PORT/ws/events`
- **方向**: 服务端 → 客户端（单向推送）
- **格式**: 每条消息一个 JSON Event 对象
- **心跳**: 服务端每 30 秒发送 ping，客户端回 pong
- **重连策略**: 客户端断连后自动重连，重连后通过 `/api/events?since_sequence=N` 补偿 gap

#### 4. Web Frontend（React）

- **Network Topology Graph**: Agent 网络拓扑图，节点=Agent，边=消息通道
- **Message Timeline**: 右侧消息流时间线，按时间排列的消息流
- **Agent Status Panel**: 底部 Agent 详情面板
- **Dashboard Stats**: 顶部全局统计条

##### Zustand Store 结构

```typescript
interface AgentStore {
  // 数据
  agents: Map<string, AgentCard>       // Agent 列表（按 ID 索引）
  messages: Message[]                   // 消息历史（FIFO，上限 1000 条）
  events: Event[]                       // 事件历史（FIFO，上限 500 条）
  topology: {                           // 网络拓扑
    nodes: AgentCard[]
    edges: { from: string; to: string; messageCount: number }[]
  }
  stats: {                              // 全局统计
    agentCount: number
    messageCount: number
    activeAgents: number
    errorCount: number
    uptime: number
  }
  lastSequence: number                  // 最后收到的事件 sequence（用于重连补偿）

  // UI 状态
  selectedAgentId: string | null        // 当前选中的 Agent

  // Actions
  handleEvent: (event: Event) => void   // 处理 WebSocket 推送的事件
  loadSnapshot: () => Promise<void>     // 加载 REST 快照
  selectAgent: (id: string | null) => void
}
```

### 消息动画规范 (M3b)

```
触发：每当 MessageBus 产生 message_sent 事件
形态：发光圆点（光点），直径 6px

  agent.message  = #4FC3F7 (浅蓝)

  agent.response = #81C784 (浅绿)
  agent.broadcast= #CE93D8 (浅紫，向所有目标同时发射)

到达：目标节点脉冲（scale 1→1.05→1，200ms）
并发：多条消息同时发送时，光点并存（不排队）

```

### Agent 动画规范

```
状态变化：
  idle→busy / busy→idle: 节点脉冲发光（300ms）
  any→error: 节点边框红色闪烁（持续，直到恢复）
  any→shutdown: 节点 fade out 到 opacity 0.3 + scale 0.8

入场（spawning→ready）：
  节点从中心扩散出来，opacity 0→1 + scale 0.5→1，持续 500ms

退场（shutdown）：
  opacity 1→0.3，scale 1→0.8，灰化处理
```

### 并发模型 (Eng Review)

| 组件 | 并发机制 | 细节 |
|------|---------|------|
| Agent 消息队列 | buffered channel | buffer=100，满则触发背压 |
| Agent executor | 独立 goroutine | 每个 Agent 一个，`recover()` 捕获 panic |
| MessageBus | 同步路由 | 收到消息 → 校验 → 路由 → emit 事件（同一 goroutine 内） |
| EventStream | fan-out | 维护订阅者列表，遍历写入（MVP 同步，后续可 goroutine pool） |
| WebSocket Hub | goroutine per client | 每个客户端连接一个写 goroutine |

### 优雅关闭流程

```
1. 收到 SIGINT/SIGTERM
2. API Layer: 停止接受新的 WebSocket 连接
3. Agent Manager: 向所有 Agent 发送 shutdown 指令
4. 每个 Agent: 处理完当前消息 → 状态 → shutdown → goroutine 退出
5. 超时（5 秒）: 强制关闭未响应的 Agent
6. EventStream: 发送最后的 shutdown 事件给 WebSocket 客户端
7. WebSocket Hub: 关闭所有连接
8. HTTP Server: Shutdown(ctx)
```

## Edge Cases & Error Handling (Eng Review)

### Agent 生命周期边界

| 场景 | 处理方式 |
|------|---------|
| Agent spawn 失败（超时/panic） | spawning→error，emit `agent.error`，Manager 记录失败 |
| Agent goroutine panic | executor 用 `recover()` 捕获，状态→error，emit 事件，不影响其他 Agent |
| 发消息给不存在的 Agent | MessageBus 返回 error，emit `agent.error`（type=undeliverable） |
| 发消息给已 shutdown 的 Agent | 同上 |
| Agent 消息队列满 | 丢弃消息 + emit `agent.error`（type=backpressure） |

### WebSocket 边界

| 场景 | 处理方式 |
|------|---------|
| 客户端断连 | 从订阅列表移除，不影响 Runtime |
| 客户端重连 | 先 REST `/api/agents` 获取全量快照，再 WS 订阅，用 `since_sequence` 补偿 gap |
| 多 tab 同时打开 | 各自独立 WebSocket 连接，互不影响 |

### 前端状态边界

| 场景 | 处理方式 |
|------|---------|
| 消息列表无限增长 | FIFO 淘汰，保留最近 1000 条 |
| 事件列表无限增长 | FIFO 淘汰，保留最近 500 条 |
| 页面刷新 | REST 快照 + WS 增量，sequence number 补偿 |
