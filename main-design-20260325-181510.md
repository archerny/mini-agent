## Positioning

> **mini-agent is the real-time operating system for AI Agent networks.**

A self-built multi-Agent coordination engine (Go Runtime) + real-time visualization war-room dashboard (React),
designed for observing, understanding, and managing dynamic Agent collaboration networks. Ultimate vision: a one-person super-company —
give it a high-level goal, and the system automatically decomposes, assigns, executes, delivers, and retrospects.

## Problem Statement

Existing AI workflow platforms (n8n / Dify / Coze) are essentially static DAG pipelines — nodes execute along preset paths with no real multi-Agent dynamic collaboration. When you want to build a "one-person super-company" — multiple AI Agents playing different roles and autonomously collaborating on complex tasks — existing tools fall short. You need:

1. **Agent Runtime**: A true multi-Agent coordination engine where Agents interact dynamically, statefully, and autonomously
2. **Visualization Platform**: A real-time war-room command dashboard showing each Agent's status, message flow, and dependencies

### Specific Pain Points

- Can't see what Agents are doing — when multiple Agents run in parallel, there's no intuitive view of each Agent's current state
- Message flow is untraceable — what messages Agents exchanged, in what order, whether any were lost — it's a complete black box
- Can't locate bottlenecks — which Agent is the task stuck on? Who's waiting for whom? No global view
- Can't adjust dynamically — existing pipelines are hardcoded DAGs; Agents can't autonomously decide how to collaborate at runtime

## What Makes This Cool

**Core Differentiation: From "Pipeline" to "Collaboration Network"**

- Existing tools: Node A → Node B → Node C (preset path, static execution)
- Mini-Agent: Agent A ↔ Agent B ↔ Agent C (dynamic interaction, runtime decisions)

The visualization dashboard doesn't show a linear progress bar — it shows a **living network** — messages flowing between nodes in real-time, revealing who's waiting for whom, who's stuck, who's done, and who's talking to whom. Like a real company war room.

**Differentiating Demo**: At runtime, one Agent proactively pulls another Agent into the collaboration — something impossible in n8n/Dify.

Ultimate vision: give a high-level goal, and the system automatically decomposes, assigns, executes, delivers, and retrospects.

## Constraints

- Self-use first, not pursuing generality
- Step one is "visibility" (real-time monitoring dashboard), then "controllability" (management console)
- Tech stack: Go Runtime + React frontend (Protocol-first methodology)
- Solo maintainer, architecture must be simple and controllable

## Anti-Scope (MVP explicitly excluded)

- ❌ Agent management/creation UI (MVP is read-only, no CRUD interface)
- ❌ Authentication/permission system (local self-use, no multi-user needed)
- ❌ Persistent storage (MVP is pure in-memory, restart resets everything)
- ❌ Real LLM calls (MVP uses mock/simulated Agents to validate architecture)
- ❌ Agent configuration file format design (MVP uses hardcoded demo scenarios)
- ❌ Multi-Runtime instances / distributed deployment
- ❌ Internationalization / multi-language

## Premises

1. ✅ Core differentiation is "true multi-Agent dynamic collaboration" not "static DAG pipelines"
2. ✅ Self-built Agent Runtime, not wrapping existing frameworks, full control over coordination mechanisms
   — But can later integrate existing frameworks as the Agent's LLM call layer (e.g., calling OpenAI SDK)
3. ✅ Step one is "visibility" (real-time monitoring dashboard), not "controllability"
   — Demo scenarios must be realistic enough — not just Agents saying hello, but completing an actual small task
4. ✅ Web page and Runtime are two independent but coupled systems communicating via protocol
5. ✅ I am the first user, solving my own real pain points first
6. ✅ **Protocol-first** — design the Agent communication protocol first, then build Runtime and frontend
   — The protocol is the core asset; Runtime and frontend can be independently replaced
   — Agents in any language (Python/JS/Go) can join the network via the protocol

## Agent Communication Protocol (Core Protocol Design)

### Design Principles

- **Protocol is the system's constitution** — all components are built around the protocol
- **JSON-based** — human-readable, easy to debug, ecosystem-friendly
- **Event-driven** — all state changes propagate through events
- **Extensible** — reserved metadata/extensions fields
- **MessageBus is the Single Source of Events** — Agents just send messages; MessageBus intercepts and automatically produces sent/received/error events. Agent code is unaware of EventStream, fully decoupled.

### Agent Lifecycle States

```
spawning → ready → idle ←→ busy → completed → idle
  ↓                 ↓        ↓
 error            error    error
                    ↓        ↓
                 shutdown  (recoverable → idle)
                           (non-recoverable → shutdown)
```

Complete state transition table:

| Current State | Can Transition To | Trigger Condition |
|--------------|------------------|-------------------|
| `spawning` | `ready` | Initialization successful |
| `spawning` | `error` | Initialization failed (timeout/config error/panic) |
| `ready` | `idle` | Auto-transition (ready is transient, meaning "init complete, entering idle for first time") |
| `idle` | `busy` | Received message/task |
| `idle` | `error` | Internal error |
| `idle` | `shutdown` | Received shutdown command |
| `busy` | `completed` | Current task completed |
| `busy` | `error` | Task execution error |
| `completed` | `idle` | Auto-transition (completed is transient, for event system capture) |
| `error` | `idle` | Recoverable error, Agent self-heals or gets reset |
| `error` | `shutdown` | Non-recoverable error |
| `shutdown` | (terminal) | Agent is shut down, no further transitions |

### Message Format

```json
{
  "id": "msg_uuid_v7",
  "type": "agent.message | agent.request | agent.response | agent.broadcast",
  "from": "agent_id",
  "to": "agent_id | *",
  "correlation_id": "original_msg_id (for request/response pairing)",
  "timestamp": "2026-03-25T18:15:10.000Z",
  "payload": {
    "content_type": "text | json | tool_call | tool_result",
    "content": "..."
  },
  "metadata": {}
}
```

### Message Types

| Type | Purpose |
|------|---------|
| `agent.message` | Regular message between Agents |
| `agent.request` | Agent sends a request to another Agent, expecting a reply |
| `agent.response` | Reply to a request, correlated via correlation_id |
| `agent.broadcast` | Agent broadcasts a message to all Agents |

### Agent Events (System-level)

Events are automatically produced by **MessageBus** and **Agent Manager**; Agent code does not directly emit events.

```json
{
  "id": "evt_uuid_v7",
  "type": "event_type",
  "sequence": 12345,
  "agent_id": "agent_id",
  "timestamp": "2026-03-25T18:15:10.000Z",
  "data": { "...event-specific data..." },
  "metadata": {}
}
```

**New `sequence` field**: Globally incrementing sequence number, used by frontend to detect gaps and compensate upon reconnection.

| Event Type | Producer | Trigger | Data Content |
|-----------|----------|---------|--------------|
| `agent.spawned` | Agent Manager | Agent created and initialized | `{name, role, capabilities}` |
| `agent.state_changed` | Agent Manager | Agent state change | `{prev_state, new_state, reason}` |
| `agent.message_sent` | MessageBus | Agent sends message (produced after MessageBus interception) | `{message}` |
| `agent.message_received` | MessageBus | Agent receives message (produced after MessageBus delivery) | `{message}` |
| `agent.error` | MessageBus / Agent Manager | Agent encounters error | `{error, stack, recoverable}` |
| `agent.shutdown` | Agent Manager | Agent shuts down | `{reason, exit_code}` |
| `network.topology_changed` | MessageBus | Network topology change | `{change_type: "agent_joined|agent_left|link_created|link_removed", details}` |

**Note**: `agent.message_sent` and `agent.message_received` data fields contain the complete Message object. MVP accepts data redundancy for simplicity. Can later be changed to only include `message_id` reference.

### Agent Capability Declaration (Agent Card)

```json
{
  "id": "agent_id",
  "name": "researcher",
  "role": "Research Agent — responsible for information gathering and analysis",
  "capabilities": ["web_search", "summarize", "analyze"],
  "accepts": ["agent.request", "agent.message"],
  "status": "idle",
  "metadata": {
    "model": "gpt-4",
    "max_concurrent_tasks": 3
  }
}
```

### Message Handling Rules (MessageBus)

| Scenario | Handling |
|----------|---------|
| Normal point-to-point message | Route to target Agent's message queue, emit `message_sent` + `message_received` |
| Broadcast message (`to: "*"`) | Fan-out: iterate all Agents and deliver individually, emit `message_received` for each |
| Target Agent doesn't exist | Return error, emit `agent.error` (type=undeliverable) |
| Target Agent already shutdown | Same as above |
| Target Agent's message queue full | Drop message, emit `agent.error` (type=backpressure) |
| Invalid message format | Reject, emit `agent.error` (type=invalid_message) |
| Message payload too large | Reject (limit 1MB), emit `agent.error` (type=payload_too_large) |
| Broadcast storm | Rate limit: max 10 broadcasts per Agent per second |
| `agent.response` with invalid correlation_id | Treat as regular message (non-blocking) |

## Approaches Considered

### Approach A: Go Runtime + React (Original design doc approach, not chosen)

Start building the Runtime engine directly, define protocol later.

- Pros: Faster to get running
- Cons: Protocol becomes an afterthought patch, easily couples Runtime and frontend
- Effort: L
- Completeness: 7/10

### Approach B: Go Runtime + Embedded Frontend (HTMX, not chosen)

Use Go embed + HTMX instead of React, pursuing a true single binary.

- Pros: Ultimate simplicity, zero Node.js dependency
- Cons: HTMX can't handle complex Agent network topology visualization
- Effort: M
- Completeness: 5/10

### Approach C: Go Runtime + React + Protocol-first (✅ Chosen)

Design the Agent communication protocol first, then build both Runtime and frontend around the protocol.

- Pros: Protocol is a core reusable asset; Agents in any language can connect; frontend naturally knows how to render
- Cons: More time spent upfront designing protocol (but cost is very low)
- Effort: L
- Completeness: 9/10

## Recommended Approach

**Go Runtime + React Frontend (Protocol-first)**

```
│  │  │ Manager │  │ Bus      │  │ Stream     │  │   │
│  │  │         │  │ (event   │  │            │  │   │

│  │  │  Agent A    Agent B    Agent C       │    │   │

│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**Key Design Decisions**:
1. **Agent = struct + goroutine** (not Agent = goroutine) — Agents can hibernate/resume/migrate
2. **MessageBus = Single Source of Events** — Agents are unaware of the event system, fully decoupled
3. **WebSocket unidirectional push** (MVP) — bidirectional command channel capability reserved

### Core Modules

#### 1. Agent Communication Protocol (Core Protocol Layer)

- **Message Format**: Unified JSON message protocol (see protocol design above)
- **Event System**: MessageBus + Agent Manager automatically produce events; Agents are unaware
- **Agent Card**: Agent capability declaration for discovery and routing

#### 2. Agent Runtime Engine (Go)

- **Agent struct**: Stateful Agent entity containing ID, role, state machine, message queue
- **goroutine executor**: Agent's execution engine, responsible for processing messages from the queue. `recover()` catches panics.
- **Message Bus**: Protocol-based message routing, supporting point-to-point, request/response, and broadcast. **The sole producer of message-related events**.
- **Event Stream**: Ordered stream of all events (with sequence numbers), for API layer to push to frontend
- **Agent Manager**: Manages Agent lifecycle. **The sole producer of lifecycle events**.

#### 3. API Layer (Go)

- **Protocol WebSocket** (`/ws/events`): Pushes Event Stream to frontend in real-time (unidirectional)
- **REST API**: Retrieve current network snapshot, Agent list, message history
- **CORS + dev proxy**: Support independent frontend/backend development

##### REST API Endpoints

| Method | Path | Purpose | Response |
|--------|------|---------|----------|
| GET | `/api/agents` | Get all Agents (with Agent Card) | `AgentCard[]` |
| GET | `/api/agents/:id` | Get single Agent details | `AgentCard` |
| GET | `/api/messages` | Get message history (supports `?limit=N&before=timestamp`) | `Message[]` |
| GET | `/api/events` | Get event history (supports `?since_sequence=N&limit=N`) | `Event[]` |
| GET | `/api/topology` | Get current network topology (adjacency list) | `{nodes: AgentCard[], edges: {from, to, message_count}[]}` |
| GET | `/api/stats` | Get global statistics | `{agent_count, message_count, active_agents, error_count, uptime}` |
| WS | `/ws/events` | Real-time event push | Event stream (JSON per line) |

##### WebSocket Protocol

- **Connection**: `ws://localhost:PORT/ws/events`
- **Direction**: Server → Client (unidirectional push)
- **Format**: One JSON Event object per message
- **Heartbeat**: Server sends ping every 30 seconds, client responds with pong
- **Reconnection Strategy**: Client auto-reconnects after disconnection, compensates gap via `/api/events?since_sequence=N` after reconnection

#### 4. Web Frontend (React)

- **Network Topology Graph**: Agent network topology — nodes = Agents, edges = message channels
- **Message Timeline**: Right-side message flow timeline, messages arranged chronologically
- **Agent Status Panel**: Bottom Agent detail panel
- **Dashboard Stats**: Top global statistics bar

##### Zustand Store Structure

```typescript
interface AgentStore {
  // Data
  agents: Map<string, AgentCard>       // Agent list (indexed by ID)
  messages: Message[]                   // Message history (FIFO, max 1000)
  events: Event[]                       // Event history (FIFO, max 500)
  topology: {                           // Network topology
    nodes: AgentCard[]
    edges: { from: string; to: string; messageCount: number }[]
  }
  stats: {                              // Global statistics
    agentCount: number
    messageCount: number
    activeAgents: number
    errorCount: number
    uptime: number
  }
  lastSequence: number                  // Last received event sequence (for reconnection compensation)

  // UI State
  selectedAgentId: string | null        // Currently selected Agent

  // Actions
  handleEvent: (event: Event) => void   // Handle WebSocket-pushed events
  loadSnapshot: () => Promise<void>     // Load REST snapshot
  selectAgent: (id: string | null) => void
}
```

### Interaction State Matrix

| State | Dashboard Bar | Topology | Timeline | Detail Panel |
|-------|-------------|----------|----------|-------------|
| **Loading** | Numbers = "—" | Skeleton + centered spinner + "Connecting to Runtime..." | 3-4 skeleton rows | Hidden |

| **Normal** | Real-time data | Living topology | Message stream scrolling | Expand on demand |

**Disconnection handling details**:
- Top banner appears in yellow/orange: "⚠️ Real-time connection lost, reconnecting..."
- Data retains last known state (not cleared)
- After successful reconnection, banner disappears, gap automatically compensated via `since_sequence`

### Message Animation Spec (M3b)

```
Trigger: Each time MessageBus produces a message_sent event
Shape: Glowing dot (light particle), 6px diameter
Color (by message type):
  agent.message  = #4FC3F7 (light blue)
  agent.request  = #FFB74D (orange)
  agent.response = #81C784 (light green)
  agent.broadcast= #CE93D8 (light purple, fires toward all targets simultaneously)
Path: Along topology edges (straight or curved), from source node to target node
Speed: 800ms to complete full path (ease-in-out)
Trail: 3px wide fading trail behind particle (opacity 100%→0%)
Arrival: Target node pulses (scale 1→1.05→1, 200ms)
Concurrency: Multiple simultaneous messages show concurrent particles (no queuing)
Degradation: When >20 messages per second, disable animation (only update message count)
```

### Agent Animation Spec

```
State changes:
  idle→busy / busy→idle: Node pulse glow (300ms)
  any→error: Node border red blink (persistent, until recovery)
  any→shutdown: Node fade out to opacity 0.3 + scale 0.8

Entrance (spawning→ready):
  Node expands from center, opacity 0→1 + scale 0.5→1, duration 500ms

Exit (shutdown):
  opacity 1→0.3, scale 1→0.8, desaturation
```

### User Journey (First-time Use)

```
Step 1: Run `make run`
  → Terminal: "🚀 mini-agent started on :8080"

Step 2: Open browser at localhost:8080
  → Dark dashboard skeleton → "Connecting..." → WebSocket connected

Step 3: System loads demo scenario
  → Empty state hint: "Demo scenario will auto-start in 3 seconds..."
  → Agents spawn in one by one (entrance animation)
  → Topology: nodes float out from center, arrange to layout positions
  → Stats bar: Agent count increments

Step 4: Agents start exchanging messages
  → Light particles flow along edges
  → Right-side timeline scrolls with new messages
  → Agent node colors toggle (idle↔busy)

Step 5: User clicks an Agent node
  → Bottom detail panel slides out (role/capabilities/status/recent messages)

Step 6: New Agent dynamically pulled in
  → New node entrance animation, new edge fades in
  → Stats bar Agent count +1
  → Timeline: "[system] Agent D joined the network"

Step 7: An Agent encounters an error
  → Node turns red + blinks
  → Timeline shows red error message

Step 8: Task completed
  → Agents go idle one by one → some shutdown (exit animation)
  → Stats bar shows final statistics
```

### Concurrency Model (Eng Review)

| Component | Concurrency Mechanism | Details |
|-----------|----------------------|---------|
| Agent message queue | buffered channel | buffer=100, triggers backpressure when full |
| Agent executor | independent goroutine | One per Agent, `recover()` catches panics |
| MessageBus | synchronous routing | Receive message → validate → route → emit events (within same goroutine) |
| EventStream | fan-out | Maintains subscriber list, iterates and writes (MVP synchronous, later can use goroutine pool) |
| WebSocket Hub | goroutine per client | One write goroutine per client connection |

### Graceful Shutdown Flow

```
1. Receive SIGINT/SIGTERM
2. API Layer: Stop accepting new WebSocket connections
3. Agent Manager: Send shutdown command to all Agents
4. Each Agent: Finish processing current message → state → shutdown → goroutine exits
5. Timeout (5 seconds): Force-close unresponsive Agents
6. EventStream: Send final shutdown events to WebSocket clients
7. WebSocket Hub: Close all connections
8. HTTP Server: Shutdown(ctx)
```

## Edge Cases & Error Handling (Eng Review)

### Agent Lifecycle Boundaries

| Scenario | Handling |
|----------|---------|
| Agent spawn fails (timeout/panic) | spawning→error, emit `agent.error`, Manager records failure |
| Agent goroutine panics | Executor catches with `recover()`, state→error, emit event, doesn't affect other Agents |
| Send message to non-existent Agent | MessageBus returns error, emit `agent.error` (type=undeliverable) |
| Send message to already-shutdown Agent | Same as above |
| Agent message queue full | Drop message + emit `agent.error` (type=backpressure) |

### WebSocket Boundaries

| Scenario | Handling |
|----------|---------|
| Client disconnects | Remove from subscriber list, doesn't affect Runtime |
| Client reconnects | First REST `/api/agents` for full snapshot, then WS subscribe, compensate gap with `since_sequence` |
| Multiple tabs open simultaneously | Each has independent WebSocket connection, no interference |

### Frontend State Boundaries

| Scenario | Handling |
|----------|---------|
| Message list grows indefinitely | FIFO eviction, keep most recent 1000 |
| Event list grows indefinitely | FIFO eviction, keep most recent 500 |
| Page refresh | REST snapshot + WS incremental, sequence number compensation |
